# 다중 사용자 서비스 확장 계획

> 현재 단일 사용자 localStorage 기반 → 다중 사용자 클라우드 서비스로의 전환 계획서
> 작성일: 2026-05-22

---

## 1. 현재 상태 요약

- **앱**: 정적 HTML/JS (Live Server로 실행)
- **데이터 저장**: 브라우저 `localStorage` (key: `jw-points-v1`)
- **한계**:
  - 기기 동기화 안 됨 (폰/PC 각각)
  - 브라우저 데이터 지우면 소실
  - 다른 사용자에게 서비스 불가
- **이번 작업으로 수정됨**: `calcTotal()`을 카운터 기반 → 히스토리 기반으로 변경하여 과거 날짜 backfill도 누적 포인트에 반영됨 ([app.js:143-162](../app.js#L143-L162))

## 2. 목표

- 본인 결혼 준비용 → 결혼 박람회/플래너 산업 사용자로 확대
- 보안 중요 (단, 결혼 포인트 수준의 민감도)
- 사용자 마찰 최소화 (가입/로그인이 단순할수록 좋음)
- 관리자(=본인) 부담 최소

## 3. 인증 방식 검토 (3가지 후보)

### 3-1. 닉네임 + 4자리 PIN ❌
- 무차별 대입 위험 (1만 조합)
- 분실 복구 어려움
- PIN 관리 부담
- **탈락**

### 3-2. Capability URL (자동 가입) △
- URL 자체가 비밀번호 역할 (`www.test.com/u/지원-x7n2p9`)
- 로그인 화면 없음, URL만 알면 접속
- 사용자 마찰 0
- **문제**: 봇/스팸 무한 가입 가능 → Rate Limit, Lazy 생성, Turnstile 등 방어 필요
- URL 슬러그를 사용자가 정하면 추측 가능해지므로 자동 보안 코드 6자 부착 필수

### 3-3. 구글폼 + 관리자 승인 (채택 ⭐)
- 사용자가 구글폼 제출 → 관리자가 시트 검토 후 일괄 승인 → URL 발급
- 봇/스팸 **구조적 완전 차단**
- 사용자 신원(네이버ID) 확보
- 결혼 포인트 서비스는 즉시성이 중요하지 않으므로 적합

## 4. 채택안: 구글폼 + 관리자 승인 + Capability URL

### 4-1. 전체 흐름

```
[사용자]                    [관리자(=본인)]              [시스템]
   │                              │                          │
   ├─ 구글폼 제출 ──────────────→ │                          │
   │  (naverID/닉네임/희망ID)     │                          │
   │                       구글시트에 응답 쌓임               │
   │                       검토 → 복사                        │
   │                              ↓                          │
   │                       관리자 페이지 ────────────────────→│
   │                       복붙 + [처리]                      │ URL 발급
   │                                                          │ DB 저장
   │                                       결과 표 ←─────────│
   │                                                          │
   ├─ 사이트 방문 → "내 URL 찾기" → naverID + 닉네임 입력 ────→│ 조회
   ←─────────────────────── 내 URL 표시 ──────────────────────│
   │                                                          │
   ├─ URL 접속 → 북마크 → 사용 시작                            │
```

### 4-2. URL 형식

```
www.test.com/u/jiwon-x7n2p9
                ↑↑↑↑↑ ↑↑↑↑↑↑
            사용자정의  자동생성 (비밀)
```

- 슬러그: 사용자가 희망 ID 입력 (영문 소문자+숫자+하이픈, 2~20자)
- 보안 코드: 시스템이 자동 부착 (6자 base62 ≈ 560억 조합)
- 슬러그 중복 OK (보안 코드가 다르면 별개 URL)

### 4-3. 사용자 입력 필드

| 필드 | 필수 | 형식 | 미입력 시 |
|---|---|---|---|
| 네이버 ID | ❌ | `[a-zA-Z0-9]{4,20}` | `null` (나중에 입력) |
| 표시 닉네임 (한글) | ❌ | 한글/영문 1~10자 | URL 슬러그로 표시 |
| URL 희망 슬러그 | ✅ | `[a-z0-9-]{2,20}` | (필수) |

### 4-4. 사용자 입장 흐름

```
1. 사이트 방문 → "신청하기" 버튼 → 구글폼으로 이동
2. naverID, 닉네임, 희망 슬러그 입력 → 제출
3. "1~2일 내 처리 후 본 사이트에서 확인 가능합니다" 안내
4. (며칠 후) 사이트 재방문 → "내 URL 찾기"
5. naverID + 닉네임 입력 → 본인 URL 표시
6. 클릭/북마크 → 사용 시작
```

→ 사용자가 입력하는 것: **폼 1번 + 조회 1번**으로 끝

### 4-5. 관리자 입장 흐름

```
1. 가끔 (1주에 한 번) 구글시트 열기
2. 새 응답 검토 (스팸/욕설 거르기)
3. OK인 행들 드래그 → Ctrl+C
4. 관리자 페이지 텍스트영역에 Ctrl+V → [일괄 처리]
5. 결과 확인: "✅ 12명 처리, ⚠️ 1명 슬러그 중복 → suffix 변경"
```

### 4-6. URL 전달 방식 비교

| 방식 | 사용자 마찰 | 관리자 부담 | 보안 | 인프라 |
|---|---|---|---|---|
| **A. 사이트 "내 URL 찾기"** ⭐ | 사이트 재방문 + naverID/닉네임 입력 | 0 (자동) | naverID+닉네임 둘 다 일치 시 노출 | 0 |
| B. 이메일 자동 발송 | 메일 확인 | 0 (자동) | 강함 | 메일 서비스 필요 |
| C. 카톡 수동 발송 | 카톡 확인 | 있음 (개별 발송) | 강함 | 0 |

→ **A 채택** (인프라 0, 관리자 부담 0)

## 5. 기술 스택

### 5-1. 백엔드: Supabase

- **선정 이유**:
  - Postgres 기반 → 통계/랭킹/관리자 페이지 구현 쉬움
  - Row Level Security(RLS) 강력 → 본인 데이터만 read/write 강제
  - 무료 티어: 50K MAU, 500MB DB, 2GB transfer
  - 결혼 박람회/플래너 산업 확장 시 SQL이 유리
- **프로젝트**: 기존 1개 외 신규 1개 생성 가능 (무료 organization당 active 2개까지)

### 5-2. 데이터 모델

```sql
-- 사용자
create table users (
  id uuid primary key default gen_random_uuid(),
  naver_id text,                          -- 네이버 ID (선택)
  nickname text,                          -- 한글 닉네임 (선택)
  url_slug text unique not null,          -- 'jiwon-x7n2p9' (전체)
  status text default 'active',           -- 'active' | 'suspended'
  created_at timestamptz default now(),
  last_seen_at timestamptz
);
create index on users (naver_id, nickname);  -- 조회용

-- 포인트 이벤트 (활동 1회 = 1 row)
create table point_events (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references users(id) on delete cascade,
  bucket text,                            -- 'limited' | 'daily' | 'monthly'
  activity_id text,
  occurred_at date,
  point int,
  created_at timestamptz default now()
);
create index on point_events (user_id, occurred_at);
```

**왜 이벤트 기반?**
- 누적/오늘/이번달 모두 `SUM(point) WHERE ...` 한 줄로 끝
- 과거 날짜 backfill 자연스러움 (이미 우리가 가는 방향)
- 통계/랭킹 쉬움

### 5-3. 프론트엔드

- 현재 정적 HTML/JS 그대로
- 호스팅: Vercel/Netlify (무료, GitHub 연동 자동 배포)
- 라우팅:
  - MVP: hash 방식 (`index.html#jiwon-x7n2p9`)
  - 배포 후: path 방식 (`/u/jiwon-x7n2p9`, Vercel rewrites 설정)

## 6. 보안 정책

### 6-1. URL 유출 방지

| 유출 경로 | 대응 |
|---|---|
| Referer 헤더 | `<meta name="referrer" content="no-referrer">` |
| 검색엔진 색인 | `<meta name="robots" content="noindex,nofollow">` + `robots.txt` |
| 서버 로그 | URL의 토큰 부분 마스킹 |
| 3rd-party 스크립트 | 분석/광고 스크립트 삽입 X |
| HTTP 접속 | HTTPS 강제 (Supabase 기본) |

### 6-2. 토큰 재발급

- 사용자가 URL 분실/유출 의심 시: 관리자에게 재발급 요청 (구글폼 별도)
- 새 보안 코드로 url_slug 갱신, 기존 URL 즉시 무효화

### 6-3. 관리자 페이지 보안

- 관리자 페이지 자체는 별도 강한 인증 필요 (Supabase Auth로 본인 계정만)
- IP 제한도 고려

### 6-4. Rate Limit (방어 심층)

- "내 URL 찾기" 조회 IP당 분당 10회 제한
- 무차별 대입 방지

- 카카오 로그인 추가
