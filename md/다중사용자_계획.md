### 0번 리스크 대응 완료
- ✅ **0-1. `find_my_url` brute force 차단** — 5분에 IP당 5회 초과 시 거부
  - 새 테이블 `rate_limits (ip, function_name, attempts, window_start)`
  - 헬퍼 함수 `check_rate_limit(fn, max, window_sec) returns boolean` (다른 함수에서 재사용 가능)
  - **도중 발견한 함정**: PostgreSQL은 함수가 `raise`하면 트랜잭션 전체 롤백 → `check_rate_limit`의 INSERT/UPDATE가 같이 사라짐. 즉 실패 호출은 카운트 안 됨 (rate limit 무력화).
    → 해결: `find_my_url`을 jsonb 반환으로 바꾸고 `raise` 제거. `{status: 'ok'|'rate_limit_exceeded'|'required'|'not_found'|'ambiguous', slug?}` 패턴.
    → **교훈**: 앞으로 "부수효과 + 에러" RPC는 raise 금지, JSON status 반환으로 통일.
- ✅ **0-5. 다중 기기 stale data** — 탭 복귀 시 `loadUserData` 자동 재호출 (30초 throttle), `visibilitychange` + `window.focus` 이벤트
- ✅ **0-7. 자정 넘김 화면 미갱신** — 5분 `setInterval`로 `recomputeCounters() + render()` (네트워크 X)
- ✅ **0-8. alert 도배 UX** — toast 컴포넌트로 교체. 좌측 컬러 바 (빨강=에러 / 초록=성공 / 파랑=정보). mutation path 6곳 교체. 차단 필요한 `confirm`은 유지.

### 부수 개선
- [app.js](../app.js) `recomputeCounters()` 도입 → `state.limited/daily/monthly`는 항상 `history`에서 derive. 기존 `resetIfNeeded()` 제거 (자정/월초 리셋 무료).

## 0-A. 추후 해야할 작업 (사용자 ~500명 도달 전 검토, 2026-05-29 보류)

> 0번 리스크 중 일부를 "지금은 안 하고 사용자 늘면 결정". 즉시 적용 여부는 미정 상태.
> ✅ `find_my_url` rate limit (0-1번): 2026-05-29 적용 완료 (별도 항목으로 위치).

### A-1. 관리자 비번 brute force 보호
`admin_bulk_create_users` 비번 틀려도 무제한 시도 가능. bcrypt 자체 throttle(~100ms)에 분산 공격 대비 추가 보호 필요.
- **방식**: 이미 추가된 `rate_limits` 테이블 + `check_rate_limit` 헬퍼 재사용. 함수 내 `perform check_rate_limit('admin_bulk_create_users', 5, 60)` 한 줄 추가.
- **예상 작업**: SQL 1줄 + 함수 재정의. 5분.
- **우선순위 격상 시점**: 관리자 페이지 URL을 누군가에게 공유했거나 노출 의심 시 즉시.

### A-2. DB 백업 자동화
무료 티어는 자동 백업 X. "전체 초기화" 사고 또는 DB 장애 시 회복 불가.
- **방식 A**: GitHub Actions 주 1회 → `pg_dump` 결과를 private repo에 commit
- **방식 B**: 수동 — Supabase Dashboard에서 SQL Editor `select * from users / point_events` → CSV export → 로컬 보관
- **예상 작업**: 방식 A는 셋업 20~30분 + workflow yaml. 방식 B는 즉시 가능.
- **우선순위 격상 시점**: 실데이터(본인 외 사용자)가 들어가기 시작할 때.

### A-3. URL 재발급 기능
500명 규모에선 (naverID, 닉네임) 중복 충돌이 가끔 발생 → `find_my_url`이 `ambiguous` 거부. 또는 사용자가 URL 분실/유출 의심 시 새 URL 필요.
- **방식**:
  - (a) 관리자 페이지에 "특정 사용자 URL 재발급" UI 추가 — 사용자가 카톡 등으로 요청 시 관리자가 신규 발급 → 기존 url_slug 무효화
  - (b) `admin_bulk_create_users` 등록 시 (naverID, 닉네임) 중복 사전 차단 → 충돌 자체 방지
- **예상 작업**: (a)는 10~15분, (b)는 5분.
- **우선순위 격상 시점**: 첫 충돌 신고 또는 URL 유출 우려 발생 시.

### A-4. (옵션) 자동 승인 룰
신청 누적되면 주 1회 수동 검토 부담 ↑. 특정 조건(예: 네이버ID 입력 + 슬러그 형식 OK) 충족 시 자동 통과로 처리량 ↑.
- **트레이드오프**: 봇/스팸 차단력 약화. 구글폼 reCAPTCHA에 의존도 ↑.
- **우선순위 격상 시점**: 주 50건 이상 신청 누적 시.

## 0. 운영 중 유의사항 / 알려진 리스크 (2026-05-29 추가)

> Phase 2 구현 직후 정리한 운영 리스크 체크리스트. 우선순위 순.

### 🔴 시급 (운영 시작 전 챙길 것)

#### 1. `find_my_url` brute force = 슬러그 유출 통로 ✅ (2026-05-29 적용 완료)
naverID + 닉네임 조합 추측으로 다른 사람 슬러그를 알아낼 수 있음. 슬러그 하나만 따면 그 사용자 **모든 R/W 가능**. 무제한 호출이라 자동화 공격이 쉬움.
- **대응**: SQL 함수 안에 시도 횟수 제한 테이블 추가 또는 Supabase Edge Function에 IP rate limit (분당 5회 정도)
- **결과**: `rate_limits` 테이블 + `check_rate_limit` 헬퍼 도입, `find_my_url`이 5분/IP/5회 초과 시 `rate_limit_exceeded` 반환

#### 2. 관리자 비번 brute force
`admin_bulk_create_users` 비번 틀려도 무제한 시도. bcrypt가 한 번 ~100ms라 자체 throttle 효과는 있지만, 모르는 사이 사용자 무더기 생성 당할 수 있음.
- **대응**: 동일하게 함수 내 시도 카운트 또는 Edge Function rate limit

#### 3. DB 백업 없음
무료 티어는 자동 백업 X. "전체 초기화" 잘못 누르거나 DB 사고 시 영영 못 살림.
- **대응**: 주 1회 SQL Editor에서 `select * from users; select * from point_events;` 결과 CSV로 export. 자동화하려면 GitHub Action에서 `pg_dump`, 또는 Supabase Pro ($25/월)

### 🟡 운영 중 마주칠 버그/논리 이슈

#### 4. 활동 max 초과 가능 (race + 다중 기기)
클라이언트의 `count >= max` 체크는 stale 데이터 기반. 두 기기에서 동시에 + 누르면 max 초과해서 저장됨. DB에 max 제약이 없음.
- **대응**: `add_point_event` RPC 안에 max 체크 추가 (단, data.json의 max를 DB로 옮기거나 인자로 받아야 함). 본인만 쓸 땐 무시 가능.

#### 5. 다중 기기 stale data ✅ (2026-05-29 적용 완료)
폰/PC 동시 사용 시 한쪽 변경이 다른쪽에 반영 안 됨. 다른 쪽에서 + 누르면 옛 카운트 기반으로 동작.
- **대응**: 페이지 visibility 이벤트로 포커스 복귀 시 `loadUserData` 재호출. 또는 Supabase realtime 구독.
- **결과**: `visibilitychange` + `window.focus` 이벤트로 refetch (30초 throttle), 5분 interval에서도 fetch 시도

#### 6. 타임존 미묘함
클라이언트 `todayKey()`는 로컬(KST), DB `created_at`은 UTC. 한국 자정 직후 입력하면 일관성 깨질 수 있음. 큰 문제는 아니지만 통계 낼 때 1일 차이.
- **대응**: 모든 날짜 계산은 클라이언트가 보낸 `occurred_at`만 신뢰. (현재 코드는 이미 그렇게 동작 — `created_at`은 메타데이터만)

#### 7. 자정 넘김 화면 미갱신 ✅ (2026-05-29 적용 완료)
페이지 켜둔 채로 자정 지나도 "오늘 적립" 카운트가 안 갱신됨. 새 클릭이 일어나야 recompute됨.
- **대응**: `setInterval`로 5분마다 `recomputeCounters() + render()` 호출. 한 줄로 끝.
- **결과**: 5분 interval 적용 완료. 추가 부수효과로 `resetIfNeeded()` 제거 가능해짐 (카운터는 항상 history에서 derive).

#### 8. 에러 UX가 alert 도배 ✅ (2026-05-29 적용 완료)
RPC 실패하면 모두 `alert()`. 네트워크 끊기면 클릭마다 팝업. 모바일에서 거슬림.
- **대응**: toast 컴포넌트로 교체 (작은 작업)
- **결과**: mutation path 6곳을 `showToast(msg, type)`으로 교체. type: error/success/info. 차단 필요한 `confirm`은 유지.

### 🟢 Phase 3 갈 때 챙길 것

#### 9. CORS 좁히기
지금 Supabase는 모든 origin 허용 (`file://` 포함). 배포 도메인 정해지면 Supabase Dashboard → API → CORS Origins에서 그 도메인만 허용.

#### 10. 매직링크 전환 준비

#### 11. 개인정보 처리

### 🔵 알면 좋은 자잘한 것

#### 12. `data.json` 변경 시 주의
활동 ID(`a01` 등) 바꾸면 기존 `point_events.activity_id`와 매칭 안 됨 → 그 이벤트들 사라진 것처럼 보임. **ID는 immutable로 두고 name/note만 수정**할 것.

#### 14. URL 노출 경로
계획서 6-1에 있는 것들 (referer, robots, 분석 스크립트 등) — 배포 전 점검.

#### 15. Supabase 무료 한도 모니터링
50K MAU / 500MB DB / 2GB egress. 대시보드에서 주기적으로 확인.

## 4. 채택안: 구글폼 + 관리자 승인 + Capability URL

### 4-1. 전체 흐름

```
[사용자]                    [관리자(=본인)]              [시스템]
   │                              │                          │
   ├─ 구글폼 제출 ──────────────→ │                          │
   │  (naverID/닉네임/희망ID)     │                          │
   │                       구글시트에 응답 쌓임               │
   │                       검토 → 복사                        │
   │                              ↓                          │
   │                       관리자 페이지 ────────────────────→│
   │                       복붙 + [처리]                      │ URL 발급
   │                                                          │ DB 저장
   │                                       결과 표 ←─────────│
   │                                                          │
   ├─ 사이트 방문 → "내 URL 찾기" → naverID + 닉네임 입력 ────→│ 조회
   ←─────────────────────── 내 URL 표시 ──────────────────────│
   │                                                          │
   ├─ URL 접속 → 북마크 → 사용 시작                            │
```

### 4-2. URL 형식

```
www.test.com/u/jiwon-x7n2p9
                ↑↑↑↑↑ ↑↑↑↑↑↑
            사용자정의  자동생성 (비밀)
```

- 슬러그: 사용자가 희망 ID 입력 (영문 소문자+숫자+하이픈, 2~20자)
- 보안 코드: 시스템이 자동 부착 (6자 base62 ≈ 560억 조합)
- 슬러그 중복 OK (보안 코드가 다르면 별개 URL)

### 4-3. 사용자 입력 필드

| 필드 | 필수 | 형식 | 미입력 시 |
|---|---|---|---|
| 네이버 ID | ❌ | `[a-zA-Z0-9]{4,20}` | `null` (나중에 입력) |
| 표시 닉네임 (한글) | ❌ | 한글/영문 1~10자 | URL 슬러그로 표시 |
| URL 희망 슬러그 | ✅ | `[a-z0-9-]{2,20}` | (필수) |

### 4-4. 사용자 입장 흐름

```
1. 사이트 방문 → "신청하기" 버튼 → 구글폼으로 이동
2. naverID, 닉네임, 희망 슬러그 입력 → 제출
3. "1~2일 내 처리 후 본 사이트에서 확인 가능합니다" 안내
4. (며칠 후) 사이트 재방문 → "내 URL 찾기"
5. naverID + 닉네임 입력 → 본인 URL 표시
6. 클릭/북마크 → 사용 시작
```

→ 사용자가 입력하는 것: **폼 1번 + 조회 1번**으로 끝

### 4-5. 관리자 입장 흐름

```
1. 가끔 (1주에 한 번) 구글시트 열기
2. 새 응답 검토 (스팸/욕설 거르기)
3. OK인 행들 드래그 → Ctrl+C
4. 관리자 페이지 텍스트영역에 Ctrl+V → [일괄 처리]
5. 결과 확인: "✅ 12명 처리, ⚠️ 1명 슬러그 중복 → suffix 변경"
```

### 4-6. URL 전달 방식 비교

| 방식 | 사용자 마찰 | 관리자 부담 | 보안 | 인프라 |
|---|---|---|---|---|
| **A. 사이트 "내 URL 찾기"** ⭐ | 사이트 재방문 + naverID/닉네임 입력 | 0 (자동) | naverID+닉네임 둘 다 일치 시 노출 | 0 |
| B. 이메일 자동 발송 | 메일 확인 | 0 (자동) | 강함 | 메일 서비스 필요 |
| C. 카톡 수동 발송 | 카톡 확인 | 있음 (개별 발송) | 강함 | 0 |

→ **A 채택** (인프라 0, 관리자 부담 0)

## 5. 기술 스택

### 5-1. 백엔드: Supabase

- **선정 이유**:
  - Postgres 기반 → 통계/랭킹/관리자 페이지 구현 쉬움
  - Row Level Security(RLS) 강력 → 본인 데이터만 read/write 강제
  - 무료 티어: 50K MAU, 500MB DB, 2GB transfer
  - 결혼 박람회/플래너 산업 확장 시 SQL이 유리
- **프로젝트**: 기존 1개 외 신규 1개 생성 가능 (무료 organization당 active 2개까지)

### 5-2. 데이터 모델

```sql
-- 사용자
create table users (
  id uuid primary key default gen_random_uuid(),
  naver_id text,                          -- 네이버 ID (선택)
  nickname text,                          -- 한글 닉네임 (선택)
  url_slug text unique not null,          -- 'jiwon-x7n2p9' (전체)
  email text,                             -- (Phase 3 매직링크 마이그레이션 대비, nullable)
  status text default 'active',           -- 'active' | 'suspended'
  created_at timestamptz default now(),
  last_seen_at timestamptz
);
create index on users (naver_id, nickname);  -- 조회용

-- 포인트 이벤트 (활동 1회 = 1 row)
create table point_events (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references users(id) on delete cascade,
  bucket text,                            -- 'limited' | 'daily' | 'monthly'
  activity_id text,
  occurred_at date,
  point int,
  created_at timestamptz default now()
);
create index on point_events (user_id, occurred_at);
```

**왜 이벤트 기반?**
- 누적/오늘/이번달 모두 `SUM(point) WHERE ...` 한 줄로 끝
- 과거 날짜 backfill 자연스러움 (이미 우리가 가는 방향)
- 통계/랭킹 쉬움

### 5-3. 프론트엔드

- 현재 정적 HTML/JS 그대로
- 호스팅: Vercel/Netlify (무료, GitHub 연동 자동 배포)
- 라우팅:
  - MVP: hash 방식 (`index.html#jiwon-x7n2p9`)
  - 배포 후: path 방식 (`/u/jiwon-x7n2p9`, Vercel rewrites 설정)

## 11. 인증 방식 단계별 전략 (확정, 2026-05-22)

### 11-1. 결론

| 단계 | 인증 방식 | 비고 |
|---|---|---|
| Phase 1 | localStorage (현재) | 본인 단독 사용 |
| **Phase 2** | **Capability URL (주소창 방식)** ⭐ | 본 계획서 4번 그대로 |
| **Phase 3** | **매직링크 (Supabase Auth) 마이그레이션 검토** | 박람회/플래너 확장 시 |

### 11-2. Phase 2에서 Capability URL을 채택한 이유

- 코드량 가장 적음 (Auth 시스템 자체가 없음 → MVP 빠름)
- 본인 + 지인 소수 규모에서는 보안 위협 거의 없음
- 이메일 입력조차 귀찮은 사용자층(어르신 등) 수용 가능
- "박람회 부스에서 QR 찍자마자 시작" 시나리오 강점

### 11-3. Phase 3에서 매직링크로 갈아탈 이유

Capability URL 방식이 사용자 수가 늘면 약점이 드러남:

- **URL = 비밀번호인데 사람들은 URL을 비밀번호처럼 안 다룸** (카톡 공유, 캡처, 브라우저 히스토리)
- **기기 이동 불편**: 6자 슬러그 직접 타이핑 또는 카톡 전송 필요
- **분실/유출 복구 부담**: 관리자에게 재발급 요청 필요
- **운영 부담**: 구글폼/관리자 승인/주간 검토 절차가 계속 누적
- **박람회/플래너 산업 표준은 이메일 기반** → 확장 시 매직링크가 정석

### 11-4. Phase 3 마이그레이션 준비 (Phase 2 단계부터 미리)

1. **`users.email` 컬럼을 nullable로 미리 추가** (위 5-2 스키마 반영 완료)
2. Phase 2 구글폼에 **이메일 필드(선택)** 추가 → 받아둘 수 있으면 받아둠
3. Phase 3 전환 시 "이메일 입력하면 매직링크로 업그레이드" UI 제공
4. 기존 url_slug 사용자는 유지하되, 매직링크 인증 후 url_slug 폐기 가능

### 11-5. Phase 2 RLS 설계 (Capability URL과 Supabase RLS 충돌 해결)

Capability URL 방식은 Supabase Auth를 안 쓰기 때문에, `auth.uid()` 기반 표준 RLS 정책이 작동하지 않음. → **RPC 함수 + 테이블 전면 차단** 패턴으로 해결:

```sql
-- 1) 테이블은 anon에게 일체 권한 없음
revoke all on users, point_events from anon;
alter table users enable row level security;
alter table point_events enable row level security;
-- (정책 안 만들면 = 전부 차단)

-- 2) 슬러그를 첫 인자로 받는 RPC 함수만 expose
create or replace function get_my_data(p_slug text)
returns table(nickname text, events json)
language plpgsql
security definer  -- 함수 안에서는 RLS 무시하고 owner 권한으로 실행
as $$
declare
  v_user_id uuid;
begin
  select id into v_user_id from users
    where url_slug = p_slug and status = 'active';
  if v_user_id is null then
    raise exception 'not found';
  end if;
  return query
  select
    (select u.nickname from users u where u.id = v_user_id),
    coalesce(json_agg(pe.*), '[]'::json)
  from point_events pe where pe.user_id = v_user_id;
end;
$$;

grant execute on function get_my_data(text) to anon;
```

**핵심**: 슬러그를 모르면 어떤 행도 못 읽음. 슬러그를 알면 그 사용자 데이터만. 전체 슬러그 목록 긁기 불가능.

포인트 추가/삭제도 같은 패턴:
- `add_point_event(p_slug text, p_activity text, p_point int, p_date date)`
- `delete_point_event(p_slug text, p_event_id uuid)`

→ 모든 RPC 함수가 **첫 인자로 슬러그를 받아 검증**하는 구조.

### 11-6. Phase 3 매직링크 전환 시 RLS 변경

매직링크로 전환되면 Supabase Auth의 `auth.uid()`가 작동하므로, RPC 함수를 점진적으로 표준 RLS 정책으로 대체:

```sql
create policy "본인 데이터만" on point_events
  for all using (auth.uid() = user_id);
```

→ 클라이언트에서 `supabase.from('point_events').select('*')` 같은 일반 쿼리가 자동으로 본인 것만 반환.
